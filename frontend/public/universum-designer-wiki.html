Universum Content Designer Reference

This guide explains how to design and create game content for Universum without writing Go code. All pieces, abilities, duration effects, traits, passives, and augments are defined as JSON files that the engine picks up automatically at startup.

No code required. Drop a new .json file in the right folder and the engine loads it on the next server start. The Content Builder GUI tool in tools/content_builder_gui.py provides a form-based interface to create these files.

Corpses Are Pieces

When a piece dies, it does not disappear from the engine — it stays on the board as a corpse (a piece with IsCorpse: true, Health: 0). Corpses keep their attached duration effects, traits, passives, augments, and defensives so designers can build "the corpse keeps doing X" mechanics.

What this means for content design:
  • Effects on a corpse tick down their duration normally, but their step chains and triggers only fire if the definition has works_on_corpse: true.
  • Abilities targeting "allies" / "enemies" / "both" do not see corpses. Use target_relation: "corpse" to explicitly target them (necromancy, corpse-explosion, raise dead).
  • The scaling sources corpse_max_health / corpse_max_energy / corpse_max_stamina read off a corpse subject. Pair with subject_type: "target" when the ability targets a corpse.
  • If an ability has a corpses cost, the consumed corpse's stats are surfaced into the execution context as corpse_consumed_max_health / _energy / _stamina / _template_id / _tags. Use these via the context scaling source for "deal damage equal to the consumed corpse's max HP" patterns.

The Step-Chain System

Almost all effects — whether they come from abilities, duration effects, or traits — are expressed as a step chain: a list of small effect steps linked by on_success and on_fail pointers. This gives you flexible branching without writing code.

Think of it like a flowchart. Each step says "do X, then if it succeeded go to step Y, if it failed go to step Z." The reserved target "end" stops execution.

Minimal Ability Step Chain

// Every useful ability needs at least: targeting → action → end
"steps": [
  {
    "id": "initial_targeting",
    "effect": "targeting",
    "parameters": { "kind": "ray", "range": 3 },
    "on_success": ["deal_damage"],
    "on_fail": ["end"]
  },
  {
    "id": "deal_damage",
    "effect": "damage",
    "parameters": { "amount": 8, "damage_type": "fire" },
    "on_success": ["end"],
    "on_fail": ["end"]
  },
  {
    "id": "end",
    "effect": "no_op",
    "parameters": {}
  }
]

Step Chain Basics

How Chains Work

Steps are executed one at a time. After each step, the engine looks at whether it succeeded or failed and follows the corresponding pointer(s).

  • Multiple step IDs in on_success / on_fail execute as a parallel branch — all are queued
  • Use "end" as a step ID to terminate a branch
  • Steps referenced must exist in the list (or be the reserved value "end")

When Steps Succeed vs Fail

EffectSucceeds when…Fails when…
targetingAt least one valid target foundNo valid targets
damageDamage was dealt (> 0)Target immune, blocked, or 0 damage
healHealing was appliedTarget already full, blocked
apply_duration_effectDE applied to at least one pieceAll targets immune
read_duration_effectValue read > 0Value is 0 or DE not present
grant_passivePassive grantedAlready has it
no_opAlways succeeds—

Passing Values Between Steps

Steps can write output values to a shared context dictionary and read them back in later steps. Use outputs on a producer step and reference the key in parameters with a $ prefix on a consumer step.

// Step 1: Read how many "bleed" stacks the target has
{
  "id": "read_stacks",
  "effect": "read_duration_effect",
  "parameters": {
    "effect_id": "bleed",
    "field": "stacks",
    "output_key": "bleed_stacks",
    "target": "target"
  },
  "on_success": ["burst"],
  "on_fail": ["end"]
},

// Step 2: Deal 3 damage × bleed stacks via scaling
{
  "id": "burst",
  "effect": "damage",
  "parameters": { "amount": 0, "damage_type": "physical" },
  "scaling": [
    {
      "source": "context",
      "key": "bleed_stacks",
      "coefficient": 3.0,
      "op": "add"
    }
  ],
  "on_success": ["end"],
  "on_fail": ["end"]
}

Scaling System

Scaling lets damage and healing amounts change based on dynamic runtime values — caster stats, stacks on a target, turn number, etc. You add a "scaling" array to a step (or the ability top-level for a default). Entries are evaluated in order, each modifying a running total.

How the math works:
1. Start with the step's base amount parameter as the running total.
2. For each scaling entry: read the source value → apply coefficient * raw + flat_value → apply that contribution to the total via op.
3. The final total becomes the effective damage/healing amount.

ScalingEntry Fields

oprequiredHow to apply: "add", "multiply", "set", "pct_add", "pct_multiply"
subject_typeoptional"caster" (default) or "target" — whose piece to read from
coefficientoptionalMultiplier on the raw value. Default 1.0 if both this and flat_value are 0.
flat_valueoptionalAdditive offset on the raw value. contribution = raw * coeff + flat
min / maxoptionalClamp the contribution before applying op
round_modeoptionalRounding after all entries: "floor" (default), "ceil", "round", "none"

Op Values

multiplytotal *= contributionScale total by a multiplier
settotal = contributionReplace total with computed value entirely
pct_addtotal += base_amount * contribution / 100Add X% of base amount. flat_value: 50 = +50% of base.
pct_multiplytotal *= 1 + contribution / 100Multiplicative % increase

Example — Damage that scales with caster's missing health

// At 20 missing HP: 5 + (20 * 0.5) = 15 damage

Conditions Reference

Conditions gate whether a step's results are applied. They're set as a conditions array on a step (or on a trigger). All conditions must pass (AND logic) for the step to count as having executed; if any fail, the step's on_fail branch is taken instead of on_success.

StepCondition Fields

FieldRequiredDescription
typerequiredThe condition type. Either a built-in type (table below) or a context-key name written by a previous step's outputs / by read_duration_effect.
operatoroptionalComparison operator. Default "equals". Supported: equals / ==, not_equals / !=, greater_than / >, less_than / <, greater_or_equal / >=, less_or_equal / <=.
valuerequiredThe value to compare against. Numeric, boolean, or string depending on the condition type.
optionaloptionalIf true, a malformed or unevaluable condition is treated as passing instead of erroring out. Useful for forward-compatible conditions.

Context Condition (fallback)

If type is not in the built-in table, it's looked up in:

  1. The current step's outputs (values written by the step's effect — e.g. affected_count after a damage step)
  2. The shared context dictionary (values written by previous steps' outputs map or by read_duration_effect)

The retrieved value is then compared against value using operator. This is how you write things like "only proceed if the previous step actually hit something":

"conditions": [
  { "type": "affected_count", "operator": ">", "value": 0 }
]
// Step passes only if the previous effect actually affected at least one piece.

Cast & Channel

An ability becomes a cast-time channel when it has a "cast" object on its top-level. The piece begins channeling on activation and resolves the ability N turns later. Cast-time abilities support charge-up, interrupt protection, while-casting buffs, and other rich interactions.

Cast Lifecycle

  1. Cast Start — player activates ability, cost paid (if cost_timing: "start"), while_casting_steps runs, cooldown may start.
  2. While Casting — the piece is in Casting state. Each owner turn decrements turns_remaining. CC, death, voluntary movement (unless allowed), or excessive forced movement may interrupt.
  3. Cast Resolve — when turns_remaining reaches 0, the ability's main steps chain executes. If target_lock_timing: "cast_resolve", the player chooses targets at this moment.
  4. Cast End — cost paid (if cost_timing: "resolve"), cooldown starts (if cooldown_timing: "resolve"), while_casting duration effects are removed.

Shared Resource Groups

Cooldowns and uses can be shared across several abilities on the same piece via independent group ids (cooldown_group on cooldowns, uses_group on uses). When one member is used, the cooldown is applied — and the use is spent — across all members of the group on that piece. The two group kinds are independent: you can share a cooldown without sharing uses, or vice-versa.

Example. Two stances twinblade_emberstrike and twinblade_froststrike both set "cooldown_group": "twin_slash" with a 2-turn cooldown — so the duelist picks one element per cooldown window; using either locks both.
Start state. start_on_cooldown / start_cooldown_turns (cooldowns) and start_uses (uses) seed the ability's runtime state at creation time. Because seeding happens at loadout start, on summon, and on form entry, a form-specific ability can begin its form already on cooldown.

Exhaust

"exhaust": { "enabled": true, "duration_turns": 1 }

When exhausted, the piece cannot use any more abilities until the exhaustion expires. Set duration_turns: 0 to exhaust until end-of-turn only.

Step Injections

Step injections let an augment add brand-new steps into the modified ability's chain, either before or after an existing step (referenced by ID), or completely replacing one. Defensives

Defensives are damage-mitigation layers that pieces can have either intrinsically (declared on the piece template) or grantable at runtime (declared in runtime/defensives/ and granted via the grant_defensive step). There are 12 defensive types covering everything from flat HP shields (ward) to dodge mechanics (evasion, phasing) to damage-to-DoT conversion (stagger).

Direct-hit pipeline (per room/defensive_processor.go):
phasing → evasion → foresight → block → resistance → absorption → armour → ward → recuperate → stagger → damage_cap → health.
DoT pipeline (DE ticks): resistance → absorption → ward → health. Phasing/evasion/foresight/block/armour/recuperate/stagger/damage_cap don't fire on DoTs.
Immunity-by-tag blocks the hit before either pipeline runs.

Auras

Auras are "while in range" effect carriers attached to a piece. While the carrier is alive, every qualifying piece within range hex tiles continuously receives the aura's effect — either an auto-applied duration effect (cheap, the recommended path) or a step chain that runs against each in-range piece (more expressive). Files go in runtime/auras/. Auras are granted via the grant_aura step and can be referenced from any source (passives, traits, DEs, hazards, abilities).

Movement-driven membership. The engine recalculates aura membership immediately when any piece moves. A piece walking into range gets the effect on the same tick; walking out removes it. No "wait until next turn" delay.

Auras — lifecycle & membership

An aura is a "while pieces are in range of a carrier, an effect attaches to them" mechanic. Authoring lives in runtime/auras/*.json (AuraDefinition in schema.go:2055).

State location

Aura state lives on the carrier, not on members. pieceState.Auras (an auraInstance slice in room/manager.go) records each grant. Members carry only the delivered effect (a duration effect, or the side-effect of a step chain).

One carrier can hold multiple aura instances simultaneously. The same aura definition can be granted multiple times if different sources grant it; each call yields a separate instance with its own RemainingTurns.

Two delivery modes

  • mode: "de" (default): the engine attaches the applied_de_id duration effect to every qualifying member when they come into range, and strips it when they leave. Stacking, immunity, conversion — everything DE-side — applies normally. Most current auras use this mode.
  • mode: "steps": the engine runs the aura's steps chain against each qualifying member every tick (per tick_timing). More expressive (one-shot effects, custom logic) but more expensive than mode=de.

Membership recalc

After any piece movement, recalcAllAuraMembershipLocked (room/aura_manager.go) walks every aura on every carrier and diffs the current member set:

  • New entrant: mode=de → call applyAuraDELocked to attach the DE; mode=steps → run the step chain once against the new member.
  • Departed: mode=de → call removeDurationEffectsByAuraInstanceLocked to strip the DE. mode=steps does nothing on departure (steps are one-shot).

This is O(P × A) per movement — fine at current content density; profile before optimising. recalcAllAuraMembershipLocked is the canonical entry point you'd watch in a debugger if "the aura stopped firing on this ally" turns out to be a membership bug.

Duration countdown

RemainingTurns on the carrier's auraInstance ticks down via tickAuraDurationsLocked. When it hits 0, revokeAuraLocked removes the aura and strips its delivered DE from every member. RemainingTurns = -1 means permanent (e.g. an aura granted by a piece template).

Target-relation & LoS

target_relation: "allies" | "enemies" | "both" | "corpse" controls who qualifies. include_self defaults to false; set it true for self-affecting auras. require_line_of_sight is opt-in (most auras don't care about LoS; opt in for vision-gated ones).

works_on_corpse

By default a carrier's auras detach when the carrier dies. Set works_on_corpse: true on the aura def to keep it firing while the carrier is a corpse. Common for necromancy/ghost-style designs.

The four shipped auras

  • inspiring_presence — Range 2, allies (excl. self), mode=de → fortify DE (stat buff).
  • guardians_bough — Range 2, allies (excl. self), mode=de → guarded DE (installs an incoming-damage redirect rule on each ally, target = $source resolving to the carrier).
  • martyrdom — Range 2, allies (excl. self), mode=de → martyrdom_protected DE.
  • siphon_wells — Range 2, enemies, mode=de → siphon_marked DE (inverse redirect — heals drain to carrier).

Open design question

The "aura wraps a DE" indirection feels heavy for simple-passive use cases. A future redesign may let auras attach a passive directly (no DE wrapper) or expose aura semantics as a step effect / targeting kind. The current model is what's shipping; redesigns are out of scope for now.

Traits, Passives & Duration Effects — when to use which

Quick map

  • Trait (runtime/traits/*.json): a labeled property a piece "has." Optional triggers, optional stat-modifiers, optional properties bag for content-specific state. Permanent by default (set duration for timed). Examples: flying, rage.
  • Passive (runtime/passives/*.json): same shape as a trait but also supports ability_grants, trait_grants, and the special prevent_death spec. Use when the effect grants something else to the piece rather than being a self-describing label. Examples: treant_grant_aura_on_spawn, tree_death_bloom.
  • Duration effect (runtime/duration_effects/*.json): a temporary stackable status that ticks. Has first-class stacking_type, max_stacks, conversion, per-tick steps, lifecycle steps (on_applied_steps/on_removed_steps/on_expired_steps). Examples: bleed, rage_stack, guarded.

Instance state lives in different shapes

traitInstance and passiveInstance carry InstanceID, SourceID, SourceType, CurrentDuration, and (for passives) PreventDeathChargesUsed. Neither carries a stack count. A trait/passive is either present or absent; if you want "this thing has N stacks," the engine pattern today is to dispatch a duration effect from the trait/passive and let the DE hold the stacks.

The "rage" dispatcher pattern

The shipped rage trait demonstrates the convention:

  1. Trait rage in runtime/traits/rage.json has triggers on on_damage_taken and on_damage_dealt. Each trigger's step chain calls apply_duration_effect with effect_id: "rage_stack", stacks: 1, duration: 1.
  2. DE rage_stack in runtime/duration_effects/rage_stack.json uses stacking_type: "intensify", max_stacks: 5, and a stat_modifiers entry with per_stack: true so the +energy bonus scales with stack count.
  3. Other content reads the stack count via the de_stacks scaling source — e.g. an ability deals extra damage scaled by rage_stack's current stacks on the target.

Future direction: moving stacks onto traits directly (so the trait itself carries a stack count, and per-stack stat modifiers read from the trait instance) would eliminate the dispatcher-DE pattern. model.TraitView already declares a Stacks field — it's just never populated today. This is the redesign discussion to have once the rest of the system is stable.

Stacking modes (DE only)

  • individual — each apply creates a fully independent instance, durations independent, stat-mods stack additively per instance.
  • replace — new application overwrites the existing instance (duration resets, stack count resets per applied stacks value).
  • extend — duration adds; stack count unchanged.
  • intensify — stack count adds (up to max_stacks); duration refreshed to the new value.

Defensives — pipeline & per-piece ownership

Damage flow

When a piece takes damage, the pipeline is:

  1. Consult incoming redirects (see Redirect Rules below). Damage may be split between the original target and one or more fanout targets.
  2. For the kept portion on the original target, run the defensive pipeline (runDefensivePipelineLocked in room/defensive_processor.go).

Redirect runs before defensives on purpose — a redirect aura should fire whether or not the ally has ward. Each fanout target then runs its own defensive pipeline independently.

Direct-hit pipeline (canonical order)

Phasing → Evasion → Foresight → Block → Resistance → Absorption → Armour → Ward → Recuperate → Stagger → Damage Cap → Health

  • Phasing — chance to make the hit not land at all (cooldown-gated; "replace" stacking, first instance wins).
  • Evasion — chance to dodge; chance grows by gain_per_miss on misses, resets on a successful dodge ("replace" stacking).
  • Foresight — pre-emptive heal scaled from a scaling expression. Does not reduce damage; just heals before damage applies. "Add" stacking (all instances heal).
  • Block — chance to apply flat + percent reduction. "Add" stacking (each instance rolls independently).
  • Resistance — percent reduction, capped at 90% total across all instances. "Add" stacking with diminishing-returns combine.
  • Absorption — flat reduction. "Add" stacking.
  • Armour — per-stack flat and/or percent reduction; consumes one stack per hit. "Sequential" stacking (temporary armour instances consumed before permanent).
  • Ward — temporary HP pool that absorbs damage 1:1 until depleted. "Sequential" stacking (temporary ward consumed before permanent).
  • Recuperate — schedules a heal-over-time DE on the defender sized as a percent of post-ward remaining damage. Does not reduce remaining; observation-only at this stage.
  • Stagger — converts a percent of remaining damage into a DoT instead of immediate damage.
  • Damage Cap — clamps remaining damage to a maximum per hit.
  • Health — whatever is left lands as HP loss.

DoT sub-pipeline

Damage-over-time ticks (from any DE applying damage on a tick) use a stripped-down pipeline:

Resistance → Absorption → Ward → Health

Phasing, evasion, foresight, block, armour, recuperate, stagger, and damage_cap do not apply to DoTs. This is enforced by the isDot flag inside runDefensivePipelineLocked — designers don't need to opt in or out.

Bypass flags

The damage step effect supports per-defensive bypass flags (bypass_ward, bypass_armour, bypass_resistance, etc.) plus a bypass_mitigation super-flag that skips the whole pipeline. Use sparingly — they bypass every instance of the named defensive, not just the temporary ones.

Per-piece ownership

Each DefensiveInstance belongs to exactly one piece. There is no shared defensive primitive — N pieces cannot consume from a single ward pool. The Treant's "shared ward" effect is implemented as a redirect aura: damage from allies redirects to the Treant, who then runs its own ward against the redirected damage. Conceptually similar to shared ward, but the mechanism is one-piece-takes-the-hit.

Twelve types

ward, armour, resistance, absorption, immunity, evasion, phasing, block, foresight, stagger, damage_cap, recuperate. Each has different fields (see the existing Defensive Types reference). Stacking behaviors (add / sequential / replace / individual) determine how multiple grants combine.

Granting at runtime

grant_defensive, revoke_defensive, and suppress_defensive step effects (see Step Effect Inventory) install / remove / temporarily disable defensives on the resolved target. Override params let you specialize the granted defensive (e.g., apply a resistance to a specific damage type at the moment of granting).

Redirect Rules — install & consume

A redirect rule says "when this piece takes damage (or heal) matching these filters, send a fraction to a different piece." Implementation in room/redirect_manager.go.

Install (the redirect step)

The redirect step effect installs a rule on the resolved target. Owner-instance metadata (which DE / passive / trait / aura the rule belongs to) is read from context keys (__duration_effect_instance_id, __aura_instance_id, etc.) so the rule auto-cleans when the owning instance is removed.

Owner-source piece (resolves the $source token) is preferentially read from __duration_effect_caster_id, then __aura_carrier_id, then opts.CasterPieceID. For an aura-applied DE, this resolves to the aura carrier — exactly right for a "redirect from members to the carrier" pattern.

Consume (the damage path)

Before defensives, consumeIncomingDamageRedirectsLocked consults the target's IncomingRedirects slice. Each rule that matches op-type, direction, and filters contributes a fanout entry. The original target's "kept" portion is multiplied by the first matching rule's keep_original_percent only — subsequent rules don't compound on the kept side. Fanout entries each re-enter applyDamage with an inRedirect flag set, preventing further redirect at the next hop.

Token grammar for redirect_to

  • $source — the rule owner's source piece (typically the aura carrier or DE caster).
  • $attacker — the source of the inbound event (often used for thorns).
  • $lowest_health_ally_in_range:N — ally of the rule owner with the lowest current HP within N hexes.
  • $nearest_enemy_in_range:N — closest enemy of the rule owner within N hexes.
  • $random_enemy_in_range:N — deterministic random pick of an enemy within N hexes (replay-stable).
  • $allies_on_connected_hazard_chain:TAG — every alive ally on a tile in the rule owner's chain-BFS for the named hazard tag, equally divided.

Filters

damage_type_filter (list) — restrict to the named damage types. source_relation_filter (any|ally|enemy|self) — restrict to incoming events from sources matching the relation to the rule-owning piece.

Auto-cleanup & loop guard

pruneRedirectRulesForInstanceLocked walks every piece and removes rules whose OwnerInstanceID matches a vanishing DE/passive/trait/aura instance. The per-piece inRedirect flag is set during redirect-fanout recursion to prevent infinite loops; rules on a piece with inRedirect=true are skipped.

Worked example: guarded

The guardians_bough aura on a Treant applies the guarded DE to allies in range. guarded.on_applied_steps installs a redirect rule on the ally: op_type=damage, redirect_to="$source", redirect_percent=1.0, keep_original_percent=0.0. When an enemy attacks the ally, the rule fires: ally takes 0, Treant takes 100% (through the Treant's own defensive pipeline).

Targeting Kinds & Tile Filters

Kinds

  • ray / single — single tile / piece at the end of a ray. Aliases.
  • projectile — line-of-sight line from caster; stops at first hit unless piercing.
  • line — straight line of N tiles; affects every piece in that line.
  • circle — disk AoE; radius required.
  • ring — hollow circle.
  • cone — directional spray; cone_mode picks the shape.
  • chain — bounce between pieces; chain_count and chain_range govern.
  • link — persistent inter-piece relationship. Sets up linkInstance records on both endpoints; link_mode selects two_piece, chain_ordered, or hub_spokes. Link is bidirectional state, not one-shot.

Tile filters

tile_filters is an ordered list of predicates a resolved tile must satisfy. Failed tiles are dropped. Predicates:

  • on_hazard_tag — tile must carry a hazard whose tags include tag.
  • not_already_hazard_tag — tile must NOT carry any hazard with tag.
  • not_occupied — no living (non-corpse) piece on the tile.
  • adjacent_to_caster_or_hazard_tag — hex-adjacent to caster OR to any tile carrying an owner-matching hazard with tag. Used by chain-hazard placement.

Each filter has an optional negate: true flag that inverts the predicate.

LoS knobs

  • ignore_line_of_sight: true — bypass per-target LoS check entirely. Use for curses, ground-targeted utility, or abilities that should pass through cover.
  • aoe_ignores_line_of_sight: true — disable the per-tile LoS filter inside an AoE shape. Use when the AoE should ignore cover within its blast radius.

Today there is no "vision through chained allies" mechanic. If you want roots/hazards to act as vision sources, use the simpler ignore_line_of_sight for now.

Hazards — placement, reveal & chain hazards

Lifecycle

Place via the place_hazard step. The hazard instance is appended to the target tile's Hazards slice with OwnerUserID set to the placing piece's owner. Lifecycle steps:

  • on_entry_steps — fire when a piece enters the tile.
  • on_exit_steps — fire when a piece leaves the tile.
  • on_turn_start_steps / on_turn_end_steps — fire each turn (filtered by trigger_when_occupied_only if set).
  • on_disconnected_steps — chain hazards only; fire when the hazard is no longer connected to its anchor.

Hidden hazards

is_hidden: true makes the hazard invisible to enemies until triggered. On first enemy entry the hazard sets IsRevealed: true before running on_entry_steps. This means reveal-then-effects is the design — entry steps still fire on the triggering piece. If on-entry seems "to just reveal," the step chain is failing silently (often: wrong effect name or invalid target resolution).

Common targeting in hazard steps

Inside a hazard's on_entry_steps, the step's target: "self" resolves to the entering piece, not the hazard itself. Match the caltrops pattern: "target": "self" with "effect": "damage" or "effect": "apply_duration_effect". Do not use "effect": "apply_status" — that's a Server2-era name with no engine handler.

Stacking modes

replace — new instance overwrites existing instances of the same hazard on the tile.
extend / stack — append duration to the first existing instance.
individual (default) — append a fresh independent instance.

Chain hazards (E3)

chain_anchor_tag: "caster" declares the placing piece as the anchor. chain_member_tag tags fellow chain hazards. The BFS at room/chain_bfs.go walks hex-adjacent neighbours through tiles carrying any owner-matching hazard with that member tag. Use cases: root networks that disconnect when the anchor dies (cleanup via on_disconnected_steps with {"effect": "remove_self_hazard"}), or selectors like $allies_on_connected_hazard_chain:root in redirect rules.

Other knobs

  • affects_owner — fire entry/exit on the owning player's pieces. Defaults false.
  • trigger_when_occupied_only — gate on_turn_* steps to only fire when a piece is on the tile.
  • blocks_line_of_sight — opaque hazards (e.g. smoke). Defaults false.
  • captures_tile_on_place — flip tile ownership to the placing player on every placement.
  • terrain_cost_override — change the stamina cost of entering this tile for pieces matching the relation/tag filter. Used by roots that grant free movement to specific summons.

Common gotchas

  • apply_status does not exist. Server2 had it; the new engine renamed to apply_duration_effect. Using the old name causes silent step failure.
  • target_relation values: only allies, enemies, both, corpse. Do not use self here; use target: "self" at the step level instead.
  • Step target vs target_relation: target is a per-step sentinel (self, nearest_enemy, etc.) that decides where the step acts; target_relation is the parameter inside DE/aura definitions that filters which pieces qualify.

Tag Convention: summon vs summoned

The two words look identical but mark different things; following the convention lets the client / augment / scaling layers reason about them correctly.

  • summon / summoning — tag the ability or card or step that creates a summoned piece. Used by augments that scale "summoning effects," by costs that require a summon ability tag, and by conditions that branch on "is this a summon-creating action?"
  • summoned — tag the piece itself, indicating it was created mid-game by another effect rather than placed at loadout time. The pygame loadout panel filters anything with this tag out of the piece-selection list so summons aren't selectable as starting pieces. Server endpoints still return summoned templates so future "preview what this ability creates" UIs can render them.

Today the four shipped summon pieces (flower, lashing_vine, mushroom, treant) carry the summoned tag. Summon-creating abilities (tree_summon_flower, tree_summon_treant, …) carry the summon tag. Treat the two as different namespaces when matching tags.

Resolution Order & Timing

Universum is Hearthstone-style: one player acts at a time, and the only things that interject are automatic triggers (on_damage, on_death, on_turn_start, …). There is no MTG-style stack, no priority passing, and no responses — when an effect resolves, any triggers it causes resolve immediately and deterministically, then control returns. This section documents the order the engine already follows so content authors can predict interactions.

Turn phases

Each turn moves through start → main → end. The player only acts during main; the start and end phases run engine bookkeeping automatically.

Turn-start order

At the active player's turn start, for each of their pieces: exhaust countdown, summon-expiry (an expired summon dies here), ability-runtime refresh, and stamina/energy reset + per-turn counter reset. Then, in this fixed order:

  1. Defensive state tick (ward replenish, armour stack gain).
  2. Duration-effect ticks — owner_turn_start then any_turn_start.
  3. Passive duration ticks, then trait duration ticks (same two timings).
  4. Aura step-mode ticks, then aura duration decrement.
  5. Cast-time tick (in-progress channels advance / resolve).
  6. on_turn_start triggers dispatched for the active player's pieces (board order).
  7. Biome / mana refresh (owned biomes refill) and the per-turn free biome placement is restored.
  8. Hazard step-cooldown tick, then hazard on_turn_start_steps.

Turn-end order

  1. Duration-effect, passive, and trait end_of_turn ticks.
  2. on_turn_end triggers for the active player's pieces (board order).
  3. Hazard on_turn_end_steps.
  4. Chain-hazard sweep (hazards that lost connection to their anchor this turn run their on_disconnected_steps).
  5. Advance to the next player; turn number increments; that player's turn-start order (above) runs.

Trigger dispatch

When an event fires (damage, heal, death, move, cast, …) the engine collects the affected piece(s) and, for each piece in turn, runs its reactions in this order: duration-effect triggers → passive triggers → trait triggers. Triggers can cause further events (a thorns passive deals damage, which can kill, which fires on_death…); this recursion is bounded by a depth cap of 10 — beyond that, deeper triggers are silently skipped to prevent infinite loops. Corpses only fire attached DE/passive/trait/aura reactions when the source opts in via works_on_corpse.

Damage, death & summon timing

Within one effect, damage/heal is applied per resolved target in target order. A piece that reaches 0 health dies immediately (becomes a corpse, on_death dispatched) before the next target is processed, so order can matter for chains. Redirected portions (links, thorns, share-pain) fan out before the kept portion lands on the original target. Summoned pieces with an expiry die at the start of their owner's turn, through the normal death pipeline.

Combat Log & Tooltips

The engine emits a detailed, MMO-style combat log and an on-demand ability tooltip preview. Both are server-computed; the client only renders.

Combat log

Effect applications emit a categorized, name-enriched entry. Each entry carries the raw IDs (for linking) and server-resolved display names — actor_name, target_name, source_name (the effect's origin piece, e.g. who applied a DoT), ability_name, effect_name — plus a category and a significant flag for client-side filtering/highlighting. Categories: damage, healing, status, aura, defensive, resource, control, death, board, cast, system.

Currently emitted events include: damage, heal, piece death, prevent-death, duration-effect applied, cast lifecycle (cast_start / cast_finish / cast_fail / cast_interrupt), summon, and biome placement. (Coverage continues to broaden toward "log everything" — finer periodic ticks and aura/defensive lifecycle events are being added.)

Delivery & history

The room-state response includes a rolling recent-50 window (recent_combat_log), already visibility-filtered: entries that involve a stealthed piece the viewer cannot currently see are omitted (entries about pieces that have since died are historical and still shown). The full match log is fetched on demand:

GET /v1/game/rooms/{roomID}/combat-log/history?since=<seq>&limit=<n>

Dynamic ability tooltips

A tooltip preview is computed live, with the piece's augments applied and scaling evaluated against current state, so the numbers reflect the real board at the moment of the request:

GET /v1/game/rooms/{roomID}/pieces/{pieceID}/abilities/{abilityID}/tooltip

It returns, per scalable output of the ability, the base value, the final scaled value, and the ordered scaling vectors (source label, op, contribution) that produced it — plus a ready-to-show formatted string such as "Deal 18 fire damage (12 base, +50% Spell Power)". Any piece the requester can currently see may be previewed (visibility-checked).

Tags, Keywords & Mana

These three areas are intentionally soft, evolving conventions rather than hardcoded rule systems. They are documented here as they currently work; they will be formalized once the shape of the content settles.

Tags (a convention, not an enum)

Tags are free-form strings used for matching, not a fixed registry: augments match abilities/traits by required/excluded tags, targeting tile-filters and scaling entries can filter by tag, and the summon / summoned namespaces distinguish summon-creating actions from summoned pieces. Author tags consistently, but expect the vocabulary to grow with the content.

"Keywords"

There is no keyword-preset macro layer yet. The expressive knobs are the CastSpec and TargetingSpec fields (cast time, target-lock timing, interrupt config, AoE shapes, link modes, …). Compose them directly on the ability. A blessed set of named presets may be added later, once recurring patterns are clear.

Mana & economy (current model)

  • Biomes placed on controlled tiles generate colored mana, one color per the 8 damage types. Each biome has an energy pool that refills at the start of its controller's turn.
  • Activating a piece spends colored mana (the piece's activation cost). Generic/colorless costs may be paid with any color.
  • Abilities additionally cost energy (refreshed each turn) and may cost stamina, health, or corpses.
  • One free basic biome placement is available per turn (modifiable by rules/effects).

This pass documents the model as-is; no economy mechanics were changed.

Authentication (JWT + JWKS)

  • Tokens. auth_service issues an access + refresh JWT pair signed with Ed25519 (EdDSA), audience universum-api-v1. refresh rotates the pair.
  • Validation at the edge. The gateway validates the Authorization: Bearer <token> header on every protected route against the auth service's published keys at GET /.well-known/jwks.json (fetched and cached, periodically refreshed), then forwards the request upstream.
  • No user tokens past the gateway. Internal services trust the gateway-forwarded request; service-to-service calls (matchmaking → game) authenticate with a shared X-Service-Token instead of a user JWT.

Gateway responsibilities

  • Allowlist reverse proxy. Only explicitly-registered routes are forwarded (see API Endpoints). Anything else 404s at the gateway even if the upstream service implements it.
  • CORS. Configurable allowed origins; methods GET / POST / PUT / DELETE / OPTIONS; credentials allowed.
  • Rate limits (per client, sliding window): auth endpoints 30/min, other protected endpoints 120/min, WebSocket upgrades 20/min.
  • Request tracing. Generates / propagates X-Request-ID across the call chain.
Known allowlist gaps. The gateway proxy allowlist does not currently include GET /v1/game/hazards/templates or GET /v1/game/rooms/{roomID}/pieces/{pieceID}/augments — both exist on game_service but would 404 through the gateway (the same class of bug that previously affected combat-log / tooltip routes). Flag when wiring a client that needs them.

Match → game-room flow

  1. Two clients connect to /v1/matchmaking/ws and queue.
  2. On a pair, matchmaking calls game's POST /internal/rooms/bootstrap (with X-Service-Token) to create the room with both loadouts.
  3. Clients POST /v1/game/rooms/join, then drive the game with the action endpoints (move / activate / cast / place biome / end-turn).
  4. Clients poll authoritative state via the reconnect heartbeat (the response carries the room state), which also keeps the seat alive.

Reconnect / heartbeat model

Each seat has a disconnect budget. The client sends POST /v1/game/reconnect/heartbeat on an interval and the response carries the current room state. If a client drops, disconnect starts the budget countdown; rejoin restores the seat (carrying the remaining budget across reconnects); forfeit ends the match. Exhausting the budget auto-forfeits.

Persistence & observability

  • Persistence. auth_service / profile_service persist users + loadouts (SQLite in dev, Postgres in production). Game-room state lives in memory in game_service for the room's lifetime.
  • Health. Every service exposes GET /health; matchmaking also exposes GET /queue/stats.
  • Logging. Structured logs via the common observability package, under each service's logs/ directory.

Worked Example — Tree of Life

The Tree of Life is the most complete piece authored to date and exercises nearly every system shipped in the engine gap-closing sprint: summons, a hazard-based root network, damage / heal redirect, auras, accumulator duration effects, prevent-death form transforms, and on-spawn / on-death triggers. This section walks the whole kit: every mechanic is described, then followed by its actual JSON.

Read it as a loop. Grow roots → roots capture tiles and become free-movement terrain for your summons → summon Flowers / Treants onto roots → the tree shares its healing across everything standing on the root chain. When the tree dies it doesn't disappear: it blooms into the Tree of Death with a darker, aggressive summon roster.

The piece

An immobile origin piece: 100 HP, 10 energy, activation cost 1 nature. It never moves (is_immobile), so its whole game is grown outward through roots and summons. It has two ability roles — grow the network (Place Root) and populate it (Summon Flower / Treant) — plus two passives: Roots Drink (heal sharing) and Death Bloom (the transform). The single death form, Tree of Death (50 HP), keeps Place Root but swaps its summon roster to Lashing Vines and Spore Mushrooms and gains Resurgent Roots.

The root network — Place Root + the Root hazard

Place Root grows a root on a tile adjacent to the tree or to any existing root, capturing the tile. It costs 1 energy and 2 health (the tree literally spends itself to expand), and runs on a 3-charge uses pool that regains 1 charge each of your turns. The targeting is a ray that ignores line of sight, gated by three tile filters: adjacent to the caster or to a root-tagged hazard, not occupied, and not already carrying a root.

The Root hazard is the substrate the whole piece is built on. It is a persistent, visible hazard that:

  • Forms a connected chain — chain_anchor_tag: "caster" (the tree anchors the network) and chain_member_tag: "root" (each root extends it). Its on_disconnected_steps self-destroy any root cut off from the chain, so severing a root prunes everything past it.

Lifecycle recap

  1. Expand. Place Root each turn (paying HP) to spread a connected, tile-capturing network that stuns enemies, wards allies, and lets your summons move for free.
  2. Populate. Summon Flowers (accumulator heals) and Treants (mobile damage-soak guardians) onto your roots.
  3. Sustain. Heal the tree and Roots Drink fans that healing out to everyone on the chain; Treants soak damage off nearby allies; Flowers top everyone up.
  4. Die forward. The first lethal hit blooms the tree into the aggressive Tree of Death (Vines + Mushrooms), and Treant / Mushroom deaths leave parting heals and explosions.
  5. Rebirth. Heal the Tree of Death to full and Resurgent Roots returns it to the supportive Tree of Life — restarting the loop.
